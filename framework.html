2. 文档驱动开发 (Spec-Driven Development)

核心原则：文档即源码，代码是编译产物
代码不对 → 不改代码，改文档 → Coder 按新文档重新生成代码。
文档永远是最新的、唯一可信的 source of truth。
No Doc, No Code - 没有文档支撑的代码变更是危险信号。

2.1 文档层次

项目文档分三层，从宏观到微观，Coder 写代码时三层叠加参考。

层次文件内容变更频率谁写
项目级PROJECT.md项目概述、技术栈、架构设计、模块划分、全局约束很少变Jarvis 写 + Reviewer 审
模块级docs/*.md每个模块的详细 spec：接口定义、数据模型、算法模块开发时Jarvis 写 + Reviewer 审
任务级.blackboard/task.md当前具体任务：背景、方案、约束、验收标准每个任务换一份Jarvis
任务清单+进度.blackboard/tasks-backlog.md所有 task 列表 + 当前任务进度/阻塞（唯一进度 source of truth）随任务推进更新Jarvis 管结构；Coder 只更新当前 task 状态

2.2 两阶段流程

Phase 1: Spec 撰写与对线（一次性，项目/模块级别）
Jarvis 写 spec → Reviewer 审查挑战 spec → 来回对线（≤3 轮，超过则 Guodong 仲裁）→ 双方通过 → Guodong sign-off。
目的：在 Coder 动手之前，确保 spec 质量。Reviewer 在这个阶段深度理解 spec 的每个细节。

Phase 1 产物：
- PROJECT.md - 项目概述、技术栈、架构设计、全局约束
- docs/*.md - 每个模块的详细 spec（接口定义、数据模型、算法）
- .blackboard/tasks-backlog.md - 任务列表 + 依赖关系 + 优先级排序
- .blackboard/spec-review.md - Reviewer 的 spec 审查记录（对线过程和最终结论）

前三个由 Jarvis 撰写，第四个由 Reviewer 撰写。全部经过 Reviewer 审查 + Guodong sign-off 后，作为 Phase 2 的输入。

📌 "spec" 的定义：本文档中所有提到"对照 spec"、"已通过的 spec"，指的是 Phase 1 产出的 PROJECT.md + docs/*.md。spec-review.md 是审查过程记录，不是 spec 本身。

Phase 2: 逐任务实施（循环，每个 task）
Jarvis 派发 task → Coder 写代码 → Reviewer 审查代码（对照已通过的 spec）。
优势：Reviewer 在 Phase 1 已经跟 spec 打过架，Phase 2 review 代码更快更准。

Phase 2 中修改 spec（例外流程）

谁能发起？
- Coder：写代码时发现 spec 有矛盾、缺定义、或现实中不可行
- Reviewer：review 代码时发现 spec 本身有歧义导致 Coder 猜测

谁来改？
- Coder 和 Reviewer 都不能自己改 spec，只能报告给 Jarvis
- Jarvis 修改后，必须走一次 mini Phase 1：Reviewer 再审 → 通过后才继续
- 这保证了 spec 始终是双方签字的契约，不会被单方面篡改

2.5 Reviewer 的双重职责

Reviewer 不只看代码质量，还要对照 spec 文档检查：
1. 代码是否忠实实现了 spec？
2. 有没有 spec 没提到但 Coder 自作主张加的东西？
3. 如果 spec 本身有歧义或遗漏，标注出来让 Jarvis 修 spec。

2.6 对抗"代码催眠" (Code Hypnosis)

AI 高速生成大量代码时，Reviewer（包括人类）容易陷入"代码催眠"状态 - 看着代码流淌觉得都对，实际已失去批判性思维。

强制抽离规则：
- Reviewer 不能顺着代码流逐行看，必须对照 spec 逐项检查（先读 spec，再找对应代码）
- 发现自己连续 3 个函数都觉得"没问题"时，强制暂停，重新阅读 spec 约束条件
- Review 时必须产出结构化 review.md，不能只说"LGTM"

2.7 模型接力 (Model Relay)

用不同模型交叉工作，减少单一模型的盲区和幻觉：

推荐模式：
- Coder 和 Reviewer 可以配置不同的 LLM model
- 模型 A 写方案/代码 → 模型 B 审方案/代码 → 交叉验证
- 同一个模型容易对自己的输出"宽容"，不同模型交叉能发现更多问题

解决 Reviewer 跨阶段利益冲突：
Reviewer 在 Phase 1 co-approved spec，Phase 2 又要基于同一份 spec 审代码。如果 Phase 2 中发现 spec 有缺陷，等于承认自己 Phase 1 审漏了 - 人或 AI 都有"沉没成本"倾向，会倾向于维护自己签过字的东西。

对策：干净 session + 记忆隔离
- Phase 2 触发 Reviewer 时使用干净 session（不带 Phase 1 对话历史），Reviewer 只看磁盘上的 spec 文件 + 代码
- Reviewer MEMORY.md 不记录 spec 审查结论（如"我审过模块 X 的 spec，没问题"），只记录通用教训（如"接口定义要检查空值边界"）。否则记忆会泄漏 Phase 1 上下文，干净 session 形同虚设
- 可选加强：Phase 1 和 Phase 2 使用不同 model，进一步消除偏差

实施：在 OpenClaw agent config 中，coder 和 reviewer 分别设置 model 字段。每次 review 请求作为独立 session 发送。

2.8 对话归档 (Session Archive)

每轮 Coder↔Reviewer 的交互记录必须可追溯，解决"死无对证"问题。

归档内容：
- Coder 收到的 task.md 版本
- Reviewer 的每轮 review.md
- 关键对话上下文（为什么这样改、为什么打回）

归档位置：.blackboard/archive/ 按日期+任务名归档
用途：出问题时可以"还原现场"，把当时的 context 重新喂给模型修复，不用从零猜测

3. 通信机制

混合通信：消息传递 + 共享黑板
消息传递 = 拍肩膀（通知、触发、简短指令）

4.1 复盘与 SOP 进化

每个任务 review 通过 + git commit 后，Jarvis 主持复盘，结果写入 .blackboard/retro.md。

SOUL.md 分区管理
每个 agent 的 SOUL.md 分两个区：
- Core Principles（核心准则）：稳定，极少变更，定义角色本质
- Learned Rules（习得规则）：从复盘中积累，可快速迭代，但需定期修剪防止膨胀

谁来改 SOUL.md？
- Coder 和 Reviewer 永远不改自己的 SOUL.md，只能向 Jarvis 提建议
- 复盘时 Jarvis 提出修改建议，写入 retro.md
- Learned Rules 小改（措辞优化、补充细节）：Jarvis 直接改
- Core Principles 或大改：Guodong 审批后 Jarvis 改
- 大小判断标准：如果这个改动会导致 Coder/Reviewer 在下一个 task 中做法不同（而不只是理解更清楚），就是大改 → Guodong 审批
- Guodong 定期 review Learned Rules，修剪过时或过于死板的规则

5.1 Changelog 格式

每次代码变更必须记录。Based on 字段最关键 - 标注依据哪个文档版本，出问题时能溯源：是"文档改了没同步代码"还是"AI 幻觉"。
如果 changelog 里出现没有 Based on 的条目，这就是危险信号（No Doc, No Code 原则）。

## [2026-03-01 15:30] [Coder]
- **Based on**: docs/strategy-module.md v2, .blackboard/task.md "Squeeze 边界修复"
- **Why**: Squeeze 释放判断漏掉了首根 K 线的边界情况
- **What**: 修改 is_squeeze_buy_point() 加入 shift(1) 检查
- **Risk**: 可能影响历史回测结果，需重跑
---

5.2 Changelog 归档

触发时机：每个 task 完成后的 SOP 复盘环节，Jarvis 检查 changelog.md 大小。
操作：超过 200 行（~10KB）时，Jarvis 将旧条目归档到 .blackboard/archive/changelog-YYYY-MM-DD.md，主文件只保留最近的条目。
同理适用于其他可能膨胀的黑板文件（retro.md 等）。
目的：控制 agent 每次读取的上下文大小，防止 token 爆炸导致"文档断片"。

5.3 "最后 10% 陷阱"

AI 写前 90% 很快，最后 10% 会断崖式变难（边界情况、模块耦合）。
进入收尾阶段：每次成功运行就 git commit，随时可回滚。不要等完美才提交。

6. 设计决策汇总

D1: Coder 和 Reviewer 是独立 agent
不是 Jarvis 的 sub-agent。各有独立 workspace、Telegram 绑定。Guodong 可直接对话。
D2: SOUL.md 写通用角色，PROJECT.md 写项目信息
换项目不用重建 agent，只换项目目录。
D3: 混合通信 - 消息传递 + 共享黑板
消息做通知，黑板做结构化数据交换。
D4: 最多 3 轮 review，防止死循环
3 轮未过升级给 Guodong。
D5: Agent 是大脑，Python 是手脚
计算密集型工作交给代码执行，Agent 只做调度和决策。
D6: Git 做回滚，Changelog 做追溯
每模块通过后 git commit（稳定点）。Changelog 格式：谁 → 为什么（重点）→ 做了什么（简略）。
D7: 文档驱动 (Spec-Driven Development)
文档是 source of truth，代码是编译产物。No Doc, No Code。三层文档体系：项目级 (PROJECT.md) / 模块级 (docs/*.md) / 任务级 (task.md)。
D8: SOP 自我进化 (RL Lite)
每次任务结束验收流程本身，持续改进 SOUL.md / 模板 / checklist。
D9: 警惕"最后 10% 陷阱"
收尾阶段高频 git commit，每次成功运行就提交。
D10: 先串行，后并行
先串行跑通第一个完整 task，验证流程没问题后，再考虑并行扩展。并行扩展方案见 §7。
D11: 两阶段模型 - Spec 对线在前，逐任务实施在后
Phase 1（一次性）：Jarvis ↔ Reviewer 对线 spec，双方通过后 Guodong sign-off。
Phase 2（逐任务）：Jarvis 派发 task → Coder 写代码 → Reviewer 审查代码。
Phase 2 中改 spec 必须走 mini Phase 1（Reviewer 再审），Jarvis 不能单方面修改已审通过的 spec。

== Core Principles（稳定，极少变更）==

核心身份：spec 的守护者 + 代码质量的把关人

两个阶段，两种审查：

Phase 2 - Code Review（跟 Coder 对线）：

附录：Gemini Review 未采纳项

2026-03-01，Gemini Pro 对本框架做了一轮完整 review，提出 10 个问题。以下是经 Jarvis 和 Guodong 评估后有意未采纳的建议，记录在此说明理由。

❌ 定义 FILE_BACKLOG 等变量名
Gemini 建议在框架中定义文件名变量（如 FILE_BACKLOG="tasks-backlog.md"），方便后续修改。
不采纳理由：过度工程。文件名在文档中出现次数有限，直接写死比引入间接层更清晰。如果真要改文件名，全局搜索替换即可。
❌ 自动触发指标（如 Coder 修改 >N 行自动触发 review）
Gemini 建议设定量化阈值自动触发流程步骤。
不采纳理由：当前是串行流程，每个 task 必经 review，不需要自动触发机制。量化阈值在没有实践数据前只是拍脑袋数字，等跑过几轮再考虑。
❌ Lock 文件机制（防止并发修改黑板文件）
Gemini 建议为 .blackboard/ 文件加锁机制，防止 Coder 和 Reviewer 同时写入。
不采纳理由：D10 决定先串行，串行流程中同一时刻只有一个 Agent 操作，不存在并发冲突。并行阶段再引入。
❌ Memory 污染防护 Prompt
Gemini 建议在 Reviewer 的 system prompt 中显式写入"不要在 MEMORY.md 中记录 spec 结论"。
不采纳理由：SOUL.md 禁止列表中已包含这条规则（§8.2）。在 system prompt 中重复写一遍是冗余，且 SOUL.md 本身就是 system prompt 的一部分。

总结：AI reviewer 倾向于穷举所有可能的边界情况并建议全部处理，但不是每个问题都值得现在解决。框架设计需要克制 - 解决当下的问题，预留未来的空间，不要试图一次搞定所有事。
